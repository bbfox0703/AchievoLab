# Architecture

## Three-Executable Design

The solution uses **process isolation** for stability:

- **AnSAM**: Lightweight game browser that only queries game ownership via `ISteamClient018` and `ISteamApps008`. Does not load achievement/stats data.
- **RunGame**: Heavy-weight per-game process spawned via `Process.Start()`. Loads achievement schemas from `UserGameStatsSchema_{gameId}.bin` files and uses `ISteamUserStats013` for editing. If this crashes, it doesn't affect the main AnSAM window.
- **MyOwnGames**: Completely independent tool using Steam Web API (HTTP REST), not Steamworks DLL.

## Steamworks Integration (steamclient64.dll)

Both AnSAM and RunGame directly load `steamclient64.dll` via P/Invoke:

1. **DLL Resolution**: Uses `NativeLibrary.SetDllImportResolver` to locate Steam installation via registry (`HKLM\Software\Valve\Steam`), then adds Steam directories to DLL search path with `AddDllDirectory()`.

2. **Interface Access Pattern**:
   - Create pipe: `CreateSteamPipe()`
   - Connect user: `ConnectToGlobalUser(pipe)`
   - Get interfaces: `GetISteamApps()`, `GetISteamUserStats()`, etc.
   - **Critical**: Use vtable-based function pointers with `UnmanagedFunctionPointer` and ThisCall convention

3. **Callback Pump**: 100ms timer calls `Steam_BGetCallback()` to process Steam callbacks (e.g., `UserStatsReceived_t`)

4. **Resource Cleanup**: Must call `ReleaseUser()` and `ReleaseSteamPipe()` in Dispose()

## Caching Architecture

Three-layer caching system optimized for responsiveness:

**Layer 1: Game List Cache** (`GameListService`)
- Location: `%LOCALAPPDATA%/AchievoLab/cache/games.xml`
- Source: `https://gib.me/sam/games.xml`
- TTL: no limit, until replace.
- A built-in `games.xml` used for failback
- `steam_games.xml` generated by **MyOwnGames**: contains game data by using Steam Web API.
- User-owned subset cached in `usergames.xml`

**Layer 2: Game Metadata** (queried via `ISteamApps.GetAppData()`)
- In-memory only, rebuilt on app launch
- Includes: titles, exe paths, launch URIs

**Layer 3: Image Cache** (`SharedImageService` + `GameImageCache`)
- Base path: `%LOCALAPPDATA%/AchievoLab/ImageCache/{language}/`
- Per-image TTL: 30 days
- Failure tracking: 7 days per language (prevents retry storms)
- Download strategy:
  1. In-memory cache check
  2. Disk cache check (with MIME validation)
  3. Language-specific download (Steam Store API → CDNs)
  4. English fallback for non-English requests
  5. Hardcoded `Assets/no_icon.png` fallback

**Concurrency Controls**:
- Global semaphore: max 10 concurrent downloads
- Per-cache semaphore: max 4 concurrent
- Per-domain rate limiting via `DomainRateLimiter` (token bucket algorithm)
- MyOwnGames uses `RateLimiterService` for Steam Web API throttling (configurable in `appsettings.json`)

## CommonUtilities Shared Library

Key shared services used across all three executables:

- `SharedImageService` - High-level image download with caching
- `GameImageCache` - Low-level disk cache + HTTP downloads
- `ImageFailureTrackingService` - Tracks failed downloads per language
- `HttpClientProvider` - Shared HttpClient singleton (thread-safe)
- `DomainRateLimiter` - Per-domain rate limiting with token bucket
- `DebugLogger` - Unified diagnostic logging

**Design principle**: No cross-cutting state. Each executable gets its own service instances.

## Achievement/Stats Editing Flow (RunGame)

When user double-clicks a game in AnSAM:

1. AnSAM spawns `RunGame.exe <gameId>`
2. RunGame creates `SteamGameClient`:
   - Validates Steam process is running
   - Connects to Steam via steamclient64.dll
   - Verifies current user is logged in (Steam ID != 0)
   - Verifies AppID matches via `ISteamUtils.GetAppId()`
3. `GameStatsService` loads schema:
   - Reads `%STEAM%/appcache/stats/UserGameStatsSchema_{gameId}.bin`
   - Parses KeyValue binary format (VDF)
   - Applies localization based on selected language
4. Requests current stats via `ISteamUserStats.RequestUserStats(gameId)`
5. Waits for `UserStatsReceived` callback
6. Queries achievement/stat values and populates UI
7. On user edit:
   - **Debug builds**: Log action but don't write to Steam
   - **Release builds**: Call Steam API, handle cascading (achievements that auto-unlock others), commit via `StoreStats()`

## Language Support

Multi-language support spans the entire data pipeline:

- **Supported languages**: en-US, en-GB, zh-TW, ja-JP, ko-KR
- **Localization points**:
  - Game titles (from Steam API or VDF schemas)
  - Achievement/stat names and descriptions (from VDF)
  - Cover images (language-specific CDN downloads)
- **Language switching**: Changes UI language, triggers image re-download from language-specific cache folders
- **Image cache isolation**: Each language has separate cache directory to avoid cross-contamination

## Language Switching & Image Loading Optimizations (AnSAM)

**Non-Blocking Language Switch**:
- When switching languages (e.g., english → tchinese → japanese), UI remains responsive
- Cached images in target language load immediately with `await Task.WhenAll()` (fast, from disk)
- Non-cached images start loading in background **without blocking** (fire-and-forget pattern)
- English fallback images display instantly while target language downloads in background

**Language Tracking** (GameItem class):
- Each game tracks which language its cover is from via `_loadedLanguage` field
- `IsCoverFromLanguage()` checks if current cover matches requested language
- `ResetCover()` clears cover path, loading flag, and language tag atomically
- On scroll (ContainerContentChanging), games with wrong-language covers are auto-reset and reloaded

**Smart Fallback Strategy** (LoadCoverAsync):
1. If requesting non-English language and English is cached:
   - Load English immediately (instant display from cache)
   - Set `_loadedLanguage = "english"`
2. Attempt to download target language in same call
3. If target language succeeds, update cover and set `_loadedLanguage = target`
4. Result: User sees English instantly, then seamless upgrade to target language

**Performance Impact**:
- Language switch completes in ~100ms (vs. 5-10 seconds before)
- No UI freezing even when 50+ images need downloading
- Images appear progressively as downloads complete
